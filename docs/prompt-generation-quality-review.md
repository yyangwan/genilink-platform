# 提示词生成质量评审

版本：v1  
状态：评审结论稿  
适用范围：`genilink-platform` 中的提示词生成链路、可见性分析入口、提示词管理页  
目标：让基础提示词更稳定、更具体，同时避免品牌名称泄漏，保证可见性分析的数据质量

## 1. 背景

可见性分析的结果，首先取决于提示词是否足够像真实用户会问的问题。

如果提示词过于泛化，后续的品牌曝光统计会出现两个问题：

- 结果容易长期偏低，甚至接近全 0
- 结果会偏离真实用户决策场景，分析结论失真

因此，提示词生成不是一个普通的内容辅助功能，而是平台核心数据生产环节。

## 2. 当前链路

当前仓库里的提示词生成入口主要有两层：

- 路由层：[`src/app/api/integration/prompts/generate/route.ts`](../src/app/api/integration/prompts/generate/route.ts)
- 规则层：[`src/lib/prompts/prompt-generation.ts`](../src/lib/prompts/prompt-generation.ts)

路由层负责从项目中取出字段，再转发给上游提示词接口。  
规则层负责判断提示词是否过泛、是否需要重新生成。

当前输入主要包括：

- `productName`
- `projectName`
- `productKeywords`
- `productDescription`
- `productUrl`

但在实际判断里，规则层主要还是围绕前三项运作，后两项没有真正进入更强的生成决策。

## 3. 现状判断

当前实现已经解决了最基础的问题：

- 不希望品牌名称直接出现在提示词里
- 不希望提示词过于空泛
- 不希望生成结果完全失去产品上下文

但它还没有形成稳定的数据生产系统，主要原因是：

1. 规则层更像启发式过滤，不像完整的生成器。
2. 路由层只负责透传输入，没有质量闸门。
3. 缺少对最终输出质量的回归测试。
4. 缺少对不同提示词意图的系统化覆盖。

## 4. 关键风险

### 4.1 产品信号太薄

如果只靠产品名、项目名和关键词，生成结果往往会停留在“名词拼接”层面。

这会导致提示词：

- 看起来有上下文
- 但没有足够强的购买意图
- 难以在真实回答中自然带出品牌

### 4.2 生成结果缺少质量门禁

当前链路没有一个统一的 `lint` 或 `score` 步骤来判断：

- 是否出现品牌名或别名
- 是否过泛
- 是否语义重复
- 是否缺少明确意图

这意味着即使输入字段正确，输出也可能不够好。

### 4.3 测试覆盖不足

现有测试主要覆盖：

- 路由转发字段是否正确
- heuristic 是否能识别泛化提示词

但还没有覆盖：

- 输出是否足够具体
- 输出是否有品牌泄漏
- 输出是否覆盖不同用户意图
- 输出是否在多轮生成中保持差异化

## 5. 建议改造方向

### 5.1 把输入提升为“产品画像”

建议先把原始字段统一抽象成一个 `product profile`，再生成提示词。

画像至少应包含：

- 产品类型
- 行业类别
- 使用场景
- 目标人群
- 购买阶段
- 对比对象
- 风险点

其中 `productDescription` 和 `productUrl` 应该参与画像构建，而不是只作为附加字段。

### 5.2 按意图桶生成

建议把提示词分成固定的意图桶，而不是继续堆模板。

推荐最少 5 类：

- 选型 / 购买
- 对比 / 替代
- 场景 / 使用
- 避坑 / 风险
- 参数 / 性能

每个意图桶都应该带有约束：

- 不能出现品牌名
- 必须包含明确场景
- 必须带有决策意图
- 不能过短、不能过泛

### 5.3 增加生成后校验

建议在生成完成后增加统一校验层：

- 品牌泄漏检查
- 泛化检查
- 重复检查
- 意图覆盖检查
- 自然度检查

这层逻辑的目标不是“纠错”本身，而是把低质量样本尽早拦掉。

### 5.4 把路由层变成质量入口

当前 [`src/app/api/integration/prompts/generate/route.ts`](../src/app/api/integration/prompts/generate/route.ts) 只负责转发。

建议后续把它升级成：

- 输入标准化
- 产品画像构建
- 上游生成调用
- 输出校验
- 失败回退 / 再生成

这样可以避免质量问题直接流入前端和后续分析环节。

## 6. 推荐的落地结构

可以把提示词生成拆成四段：

1. `buildProductProfile`
2. `generatePromptSpecs`
3. `renderPrompts`
4. `lintPrompts`

这样做的好处是：

- 规则和生成逻辑解耦
- 后面可以替换成 LLM、规则引擎或混合模式
- 测试粒度更清晰
- 调优时不会把所有逻辑锁死在一个函数里

## 7. 测试建议

建议补三类测试：

### 7.1 路由级测试

验证 `/api/integration/prompts/generate` 是否正确带入项目字段，尤其是：

- `productDescription`
- `productUrl`
- `productKeywords`

### 7.2 生成结果测试

验证输出是否满足：

- 没有品牌名泄漏
- 没有明显重复
- 不是泛化模板
- 意图明确

### 7.3 回归测试

对典型输入场景做固定样例：

- 只有行业，没有描述
- 有描述，没有关键词
- 有 URL，有描述，有关键词
- 品牌别名较多

这样可以防止后续改动把提示词质量拉回去。

## 8. 结论

当前提示词链路“能用”，但还不够稳定，也不够系统化。

如果目标是把可见性分析做成平台核心能力，那么提示词生成必须从“模板拼接”升级为“产品画像驱动 + 意图分桶 + 质量校验”的生产系统。

这不是小修小补，而是决定基础数据质量的关键层。
