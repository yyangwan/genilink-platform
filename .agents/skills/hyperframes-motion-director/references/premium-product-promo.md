# Premium Product Promo System

Use this reference when a motion video promotes a product, tool, skill, SaaS, CLI, app, API, workflow, template library, or creative system. It is especially important when the user provides a premium reference, asks for a video that feels high-end, or criticizes a draft for lacking icons, decorative elements, image resources, interaction, or sharp copy.

## First Principles

Premium product motion does not come from black backgrounds, large text, or glow alone.

It comes from visible specificity:

1. The viewer sees the product's world, not only a metaphor.
2. Each claim is backed by an artifact: command, screenshot, document, card, module, number, workflow step, output, logo, provider, or before/after state.
3. Visual density is organized into reusable components, not scattered decoration.
4. Motion makes product state change visible: input becomes output, cards fan out, modules assemble, command routes to draft, proof locks.
5. Copy names a concrete pain, concrete mechanism, concrete proof, and concrete action.

A weak cinematic promo often has one background image and several generic slogan beats. A strong product promo has a product proof ecosystem.

## Reference Teardown Pattern

When learning from a reference video, extract:

- Scene count and duration.
- Visible product artifacts per scene.
- Real assets: logos, screenshots, product images, documents, theme previews, covers, UI states, or exported outputs.
- Code-generated components: command cards, proof cards, anchored signals, chips, stat blocks, browser frames, device frames, draft cards, badges, and panels.
- Decorative system: grain, vignette, beams, floating fragments, glows, shadow plates, separators, and background fields.
- Motion grammar: what enters, what transforms, what holds, what hands off the next beat.
- Copy strategy: pain phrase, mechanism phrase, proof phrase, outcome phrase, CTA.
- Design constraints: margins, card radius, panel borders, contrast, type scale, and color restraint.

Do not copy a reference literally. Copy its production logic: proof density, asset specificity, component reuse, and motion handoffs.

## Product Proof Inventory

Before writing a product promo brief, inventory product-specific proof. If the user supplied none, inspect the project and create plausible code-native representations instead of falling back to abstract metaphors.

Look for:

- Product name, logo, favicon, app icon, mascot, or brand mark.
- CLI commands, API calls, config files, terminal output, code snippets, or workflow syntax.
- Screenshots, generated outputs, examples, templates, dashboards, exported documents, covers, charts, or before/after artifacts.
- Numbers: module count, template count, provider count, supported formats, workflow steps, saved time, version count, or benchmark result.
- Taxonomy: themes, modules, integrations, modes, providers, commands, features, or stages.
- Trust evidence: checks, validations, snapshots, review reports, export files, logs, or status badges.

For a 10-18 second vertical product promo, target:

- 1 product/brand lockup.
- 1 concrete product surface, such as a command card, UI panel, document, template stack, or output preview.
- 2-4 proof artifacts across the film.
- 4-8 chips, labels, or marks used as product taxonomy, not decoration.
- 1 recurring motion object, anchored signal, or route that connects scenes.

If fewer than three product-specific artifacts can be named, the brief must either request/supply assets, generate proof-safe representations, or acknowledge that the result will feel more like a concept film than a top-tier product promo.

## Component Library

A premium promo needs a small component library before implementation. Components must be reusable across scenes and bound to product meaning.

Useful product promo components:

- Command card: terminal, prompt, or API call with one highlighted action.
- Draft/output card: product result as a paper, article, cover, image, report, or document.
- Anchored route or signal: a workflow path, card-edge highlight, node sequence, progress strip, or CTA edge that connects specific objects; never a decorative divider.
- Stat block: one huge number with a short proof label and supporting chips.
- Module chip group: taxonomy labels such as `hero`, `toc`, `quote`, `timeline`, `cta`.
- Provider chip group: integrations or model/provider names.
- Theme/output stack: real screenshots or generated output cards fanned with perspective.
- Proof card: icon/mark, title, one-line explanation.
- Device/browser/product frame: only when it shows a real surface or believable state.
- CTA badge or action pill: the final action as a tangible button, stamp, or route endpoint.

Each component must define:

- Role in the story.
- Visual style and constraints.
- Data/content slots.
- Entry motion.
- Hold state.
- Exit or handoff.
- Rejection test.

Premium components should feel built, not drawn flat:

- Use nested shell/core structure for important cards, panels, command windows, proof cards, and CTA pills.
- Outer shell carries spacing, shadow, hairline, and radius.
- Inner core carries content, inset highlight, subtle material, and smaller radius.
- Avoid generic gray 1px borders and flat cards on a black background.
- Use borders, shadows, glow, and blur as material cues, not as default decoration.

## Icon And Decorative Element Policy

Do not ban icons and decoration. Ban meaningless icons and decoration.

Use icons or marks when they:

- Give a keyword physical behavior.
- Compress a product action.
- Make taxonomy scannable.
- Prove a workflow stage.
- Create a memorable interaction.

Acceptable marks:

- Cursor, terminal prompt, command dot, check tick, route node, upload arrow, save mark, draft stamp, image tile, provider chip, module chip, warning tag, bracket, underline, rail, lens, timeline node, document corner.

Decorative elements are allowed only as a system:

- 1 ambient texture system: grain, paper, glass, haze, or screen noise.
- 1 light system: gold beams, rim light, glow, scan, or reflection.
- 1 fragment system: paper chips, UI shards, code tokens, card shadows, or dust.
- 1 border system: panels, cards, command windows, or frames.

Reject decoration when it:

- Fills empty space without proof.
- Uses a different visual language from the product.
- Competes with title or product artifact.
- Becomes an icon pile.
- Requires labels to be understood.

## Anchored Connector Policy

Lines, paths, rails, arcs, scan strokes, and underlines are easy to overuse in black-gold promos. Treat them as product components, not decoration.

Keep a connector only when it has:

- A visible start anchor and end anchor.
- A semantic job: route, reveal, compare, focus, validate, hand off, or close.
- A motion interaction that changes product state.
- A safe-zone contract that protects the title, CTA, screenshots, and platform overlays.
- A deletion test showing what comprehension or proof is lost without it.

Reject a connector when it floats between unrelated areas, forms a pretty arc, fills empty space, or duplicates a clearer component state. In product promos, prefer attached signals: card-edge glows, route nodes, status dots, short progress strips, inspection corners, screenshot checks, chip states, cursor highlights, and CTA border lighting.

## Image And Asset Library

For product promos, image generation is not enough. Use a mixed asset library:

- Supplied official assets first.
- Existing repository assets next.
- Generated bitmap assets for atmosphere, product-safe mockups, output stacks, backgrounds, or transition plates.
- Code-generated SVG/CSS for icons, chips, panels, anchored signals, route-node sequences, command cards, and taxonomy marks.
- Screenshots or rendered local artifacts when the product's value depends on actual output.

Phase 1 must list the asset library plan:

- Existing assets to inspect.
- Generated assets to create.
- Code-native components to build.
- Screenshots or output previews to capture.
- What each asset proves.
- Which scene owns each asset.

Phase 2 must inspect and record accepted asset paths before implementation.

## Copy Sharpness

Generic copy makes premium visuals feel hollow.

Use this product promo copy ladder:

1. Pain: name the old behavior in plain language.
2. Mechanism: show how the product changes the path.
3. Proof: name a number, module, output, or workflow step.
4. Confidence: show checks, variants, providers, or generated outputs.
5. CTA: give one action that matches the product.

Prefer:

- `公众号排版 / 别再手工调`
- `一条命令 / 到草稿箱`
- `43 / 高级排版模块`
- `40 个 / 专业主题`
- `AI 写作 / 也能进流程`

Avoid:

- `让创意更高效`
- `开启全新体验`
- `提升生产力`
- `从想法到交付`
- `更智能，更高效`

If a phrase could describe any product, rewrite it.

## Motion Interaction

Every product artifact should have an interaction, not only an entrance.

Examples:

- A command card types or highlights a command, then routes to an output card.
- Chips assemble around a stat, then collapse into a module panel.
- Theme cards fan out with perspective and settle into a proof stack.
- An anchored route signal draws only when it visibly connects left steps to right output.
- A card-edge signal lights to show a command was accepted.
- A node sequence lights to show workflow order without a floating line.
- A proof card row rises in sequence and locks with a mark.
- A CTA pill receives the final underline, stamp, or cursor.

Motion review question:

If the component did not move, would the viewer still understand the product action? If yes, the motion is likely decorative. Redesign the interaction around state change.

## Premium Product Promo Gate

Block final delivery when any of these are true:

- The film uses only one abstract metaphor and no product-specific proof artifacts.
- The storyboard cannot name at least four concrete visual components for a 15s product promo.
- The asset library has no official, existing, generated, screenshot, or code-native asset decision.
- Copy is generic and could fit a different product.
- Icons or decoration are either absent when the scene needs taxonomy/proof, or present only as unrelated ornaments.
- Route lines, rails, arcs, or underlines are floating, unanchored, or only decorative.
- Motion consists of scene fades and repeated text entrance patterns.
- Snapshots look like title cards rather than product proof frames.

Target score before render:

- 20: product proof artifacts are concrete.
- 20: component library is coherent and reusable.
- 15: icon/decorative system supports meaning.
- 15: copy is specific and high-retention.
- 15: motion interactions show state change.
- 10: visual hierarchy and mobile safe zones hold.
- 5: CTA feels like the end of the product path.

Below 90 blocks final delivery for a premium product promo.
